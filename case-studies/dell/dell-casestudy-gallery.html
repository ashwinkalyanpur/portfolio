Context

🔴 The Challenge

Dell's infrastructure engineers managed load balancing, networking and DNS administration through entirely manual processes. Tasks like deploying baseline configurations, checking device compliance and pushing changes across distributed systems required specialist knowledge and were error-prone — with no single unified interface, no version tracking, and no audit trail. Identifying the root cause of non-compliant devices could take hours.

🔵 The Approach

Design a unified automation interface — Network Proteus — that gives engineers a single platform to create SNMP baselines, compare versions, verify compliance and push configuration changes via CI/CD pipelines. The UX goal was to make complex multi-step infrastructure operations feel guided and safe, with clear status feedback, rollback options, and integration with ServiceNow change management and AWX job tracking.

02 /

Four Core Modules

🗂️
MODULE 01
Create Baseline
Engineers define network baselines by entering SNMP configuration details — group name, IP addresses, subnet masks and read/write credentials. Designed to support bulk IP entry (up to 5 rows per session) with a save flow.
  • Select group name, IP address, subnet mask and RW mode
  • Add up to 5 additional IP addresses per session
  • Save baseline with version stamp
🔀
MODULE 02
Compare Baseline
Side-by-side configuration diff viewer — engineers select any two baseline versions to compare, with clear visual differencing highlighting what changed between snapshots. Critical for audit trails and compliance verification.
  • Select two baseline versions from a version list
  • View side-by-side diff highlighting changes
  • Export comparison for audit and compliance documentation
✅
MODULE 03
Device Compliance
Automated compliance checking — engineers select a baseline and trigger a full device verification sweep. Network Proteus identifies and surfaces non-compliant devices across the infrastructure, enabling rapid remediation decisions.
  • Select target baseline for compliance check
  • Click Verify Device Compliance to trigger the sweep
  • Review non-compliant device list with status indicators
🚀
MODULE 04
Push Baseline
The most complex module — engineers select regions, upload hostnames, and trigger CI/CD pipelines via AWX integration. Includes manual approval gates, scheduling, ServiceNow ticket creation, and full rollback capability with configuration file download.
  • Select region(s) and upload multi-OS hostnames
  • Auto-categorise hostnames by OS type, flag invalids
  • Create CI/CD pipeline with manual approval + scheduling
  • Track job execution in AWX, auto-create SNOW change ticket
  • Download original/changed/rolled-back config files
  • Rollback generates a separate AWX Job ID for traceability

Key Learnings

⚙️
Designing for High-Stakes ActionsPushing a baseline to live infrastructure is irreversible without rollback. Every destructive action needed clear confirmation patterns, status transparency, and a visible escape hatch — rollback and config download were not features, they were safety requirements.
🔐
Role-Based UX Changes EverythingDifferent engineers had access to different modules — the UI had to contextually show, hide and lock features based on the logged-in role. Designing for multiple permission states from the start saved major rework later.
📋
System Integrations Drive UX DecisionsAWX job IDs, ServiceNow ticket numbers and SNMP version details all surfaced in the UI. Understanding what the backend actually returned — and in what format — was essential to designing the right data display patterns.
🔀
Pipeline Status Needs Progressive DisclosureThe Push Baseline flow had 6+ steps. I learned to break these into clearly labelled stages with persistent status indicators — so engineers always knew exactly where the system was in a potentially long-running deployment job.
